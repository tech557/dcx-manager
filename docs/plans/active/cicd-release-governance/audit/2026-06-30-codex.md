# Plan Audit: cicd-release-governance

## Verdict

NOT READY

**Reason:** The architecture direction is coherent, but the draft is not yet an executable plan: it has no sprint files, no canonical requirement grounding, no carry-forward contract, and an internally inconsistent version format.

## Prior art compliance

The draft correctly cites the two relevant completed plans rather than rediscovering from scratch:

- `completed/folder-structure-v2` supplies the no-source-write discipline, archive-not-destroy precedent, carry-forward continuity model, and evidence/logging hygiene.
- `completed/requirements-system` supplies graph governance, PO sign-off before canonical mutations, append-only decision ledger behavior, `code-index` reuse, completion gates, and the requirement that future plans/sprints carry graph-backed Requirement Trace sections.

What it incorporates well: mechanical enforcement over prompt discipline, no `src/**` disruption, PO sign-off gates, append-only operational records, and archive/supersession thinking.

What it drops or leaves under-specified: the actual graph intake/sign-off before activation, per-sprint carry-forward updates, sprint-doctor close hygiene, and executable sprint files with deterministic gates. Those omissions are why the verdict is NOT READY rather than NEEDS REVISION.

## Token and mechanical-process assessment

This plan is justified only if it moves release coordination out of agent memory and into scripts, CI, CSV validation, branch protection, and promotion gates. If it remains mostly markdown plus agent instructions, it will be expensive and brittle.

Current draft size: `README.md` is 4,670 words and this audit is 1,307 words. That means a future agent that reads the README, one sprint file, carry-forward, prior output, current-state JSON, and relevant rules will likely spend roughly **12k-25k input tokens before doing useful work** on any RG sprint. Complex sprints such as RG-R3/RG-R4/RG-R7 could easily consume **35k-70k total tokens per session** once tool output, CI logs, config docs, failed attempts, and audit/log requirements are included.

| Phase | Likely agent token cost | Why | Justified? |
|---|---:|---|---|
| Revise draft into executable sprints | 20k-40k | Must turn the architecture brief into RG-R0..RG-R7 sprint files, carry-forward, trace sections, gates, and fallbacks | Yes, if it prevents repeated CI/release ambiguity later |
| RG-R0/RG-R1 governance setup | 25k-50k each | Reads rules, version model, requirement graph, PO decisions, and writes governance docs | Yes, but only once |
| RG-R2 release registry/scripts | 35k-70k | Script design, tests, validators, append-only semantics, log-index compatibility | Strongly yes; this is where later token savings begin |
| RG-R3/RG-R4 GitHub/Vercel wiring | 40k-90k each | External config, logs, failures, permissions, fallback paths, capability proof | Yes if the final gates are mechanical; not worth it as manual runbooks |
| RG-R5/RG-R6 integrations | 30k-70k each | Supabase/ClickUp/GAS setup decisions and environment safety evidence | Conditional; keep scope thin unless these are real promotion gates |
| RG-R7 dogfood | 50k-100k | End-to-end proof across source and non-source release paths | Yes; this is the proof that the machinery replaces agent judgment |

Expected one-time plan cost after revision: **250k-500k agent tokens across all RG sprints** is a realistic order-of-magnitude estimate. That is high, but defensible if it eliminates repeated human/agent release coordination and prevents accidental promotion.

Expected steady-state cost after the system exists: a normal code-change session should add only **2k-8k extra tokens** for release metadata because version assignment, preview mapping, registry validation, and promotion eligibility are mechanical. A promotion should be mostly script output plus PO approval evidence, roughly **5k-15k agent tokens**, not a full re-analysis of the release system.

Mechanical share assessment:

| Process area | Mechanical after plan succeeds? | Current draft owner | Target owner |
|---|---:|---|---|
| Source vs non-source classification | High, about 90% | Proposed script | `classify-change.sh` + CI diff |
| Version assignment | High, about 90-95% | Proposed CI merge job | Serialized GitHub Action / registry writer |
| Preview creation | High, about 95% | Vercel/GitHub proposed | Vercel branch deploy + CI registry row |
| Staging/prod promotion eligibility | High, about 85-95% | Proposed script + approval artifact | `promote.sh`, registry validator, ClickUp/approval record |
| PO approval decision | Low mechanical, by design | PO | PO remains human authority; system records and enforces |
| Requirement trace/governance | Medium-high, about 70-85% | Existing graph tools + sprint discipline | `req:*` scripts, completion gate, sprint-doctor |
| External setup/capability proof | Medium, about 40-60% | Agent/PO | Scripts can verify, but accounts/permissions remain human/external |
| Debugging failed CI/deployments | Medium, about 40-60% | Agent | Logs are mechanical; diagnosis still uses agent judgment |

Bottom line: the process can become roughly **75-85% mechanical** for routine source/non-source change handling and promotion blocking, while the first implementation is token-heavy. The cost is justified if the PO wants release safety, stable URLs, preview-per-source-change, and exact-build promotion as permanent operating constraints. It is not justified if releases remain occasional and manually supervised, because the up-front process surface is larger than the current product codebase can amortize quickly.

## Handoff quality

The README is useful as an architecture brief, but it is not enough for opencode or Codex to execute safely. The next agent would have to infer sprint steps, command syntax, output formats, fallback behavior, and PO-owned setup boundaries. The absence of sprint files means there is no durable handoff contract between RG-R0 and later sprints, and no place for Step 0/final carry-forward obligations.
