## Calibration — the intake finding (Codex #8)

Codex #8 (mechanical plain-English intake → classify → PO-confirm → contradiction/impact → technical
trace) is **directionally correct and should be explicit in RS-R0** — but per PO it is **already
conceptually locked in this session and does NOT need to be perfect in the draft.** Two concrete reasons
it's further along than #8 implies:

- **The front half already exists as a shipped rule:** `core.md §33` (added today) already requires every
  user message to be classified and typed (`sprint-execution | user-request-code | … | mixed`). The intake
  workflow is the *extension* of §33 — "is this typed message also a **candidate requirement**?" — not a
  greenfield build. **The plan must wire to §33, not reinvent classification.**
- The sign-off/supersession/relationship machinery (#8 steps 4–5) is already a README hard constraint.

→ **Fix (light):** add one hard constraint pointing RS-R0 at §33 as the intake front-end, and list the
7-step target model as a *required RS-R0 input* (cite Codex's addendum) — not as a fully-specified sprint.
Don't over-engineer the intake spec in the draft; lock the *hooks*, let RS-R0 design the mechanism.
