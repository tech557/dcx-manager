## B0.1 — Remove `isFocusIslandExpanded` from StageProvider

### Objective
Island state in the stage context causes all consumers to re-render when the Focus Island opens. Move it to local state inside FocusIsland.

### Requirements and IDs
BLD-STG-002 — island open/close must not reset stage context or other cards.

### Files to inspect
- `src/builder/stage/StageProvider.tsx`
- `src/builder/stage/stageContext.types.ts`
- `src/builder/islands/FocusIsland/FocusIsland.tsx`

### Files likely to change
- `src/builder/stage/StageProvider.tsx`
- `src/builder/stage/stageContext.types.ts`
- `src/builder/islands/FocusIsland/FocusIsland.tsx`

### Required behaviour
1. Remove `isFocusIslandExpanded` and `setFocusIslandExpanded` from `StageProvider` state and from `stageContext.types.ts`
2. Add `const [isExpanded, setIsExpanded] = useState(false)` as local state inside `FocusIsland.tsx`
3. Replace all reads of `context.isFocusIslandExpanded` and `context.setFocusIslandExpanded` in `FocusIsland.tsx` with local `isExpanded` and `setIsExpanded`

### State and data implications
StageProvider re-renders all consumers when context values change. Removing this field means Focus Island toggle no longer triggers re-renders in cards, stage, or other islands.

### Preserve-semantic boundaries
- Do not change the visual behaviour of the Focus Island
- Do not change any other StageProvider state
- Do not touch card components, readiness rules, or action files

### Acceptance criteria
```
□ grep -rn "isFocusIslandExpanded" src/builder/stage/ → 0 results
□ FocusIsland.tsx uses local useState for its expanded state
□ Focus Island opens and closes correctly
□ npm run typecheck passes — 0 errors
```

### Tests and manual validation
- Run: `npm run typecheck`
- Browser: Focus Island opens and closes; builder stage does not flash on toggle

## B0.2 — Remove `isEditorDirty` from StageProvider; add `isEditorOpen`

### Objective
Replace the semantically incorrect `focusedNodeId` usage (used to control editor column width) with a clean boolean `isEditorOpen`. Remove `isEditorDirty` from stage context.

### Requirements and IDs
BLD-STG-002, BLD-EDT-001 (editor sessions — foundation for B5).

### Files to inspect
- `src/builder/stage/StageProvider.tsx`
- `src/builder/stage/stageContext.types.ts`
- `src/builder/BuilderPage.tsx`
- `src/builder/islands/EditorViewerIsland/useEditorPanel.ts`

### Files likely to change
- `src/builder/stage/StageProvider.tsx`
- `src/builder/stage/stageContext.types.ts`
- `src/builder/BuilderPage.tsx`
- `src/builder/islands/EditorViewerIsland/useEditorPanel.ts`

### Required behaviour
1. Add `isEditorOpen: boolean` and `setIsEditorOpen: (v: boolean) => void` to StageProvider and `stageContext.types.ts`
2. Remove `isEditorDirty` and `setIsEditorDirty` from StageProvider and types
3. In `BuilderPage.tsx`: replace `hasFocusedNode` (derived from `focusedNodeId`) with `isEditorOpen` from context
4. In `useEditorPanel.ts`: add local `const [isEditorDirty, setIsEditorDirty] = useState(false)` replacing the context version; call `context.setIsEditorOpen(true)` when a node becomes active, `context.setIsEditorOpen(false)` when no active node

### State and data implications
`focusedNodeId` in StageProvider was being used solely as a boolean flag. It will remain for now (B5 will clean up the editor session model) — but `hasFocusedNode` in BuilderPage must switch to `isEditorOpen` so the editor column width is driven by the clean boolean.

### Preserve-semantic boundaries
- Editor column must still animate between 4.5rem (closed) and 25rem (open)
- Do not change any card component
- Do not change readiness.rules.ts or action files

### Acceptance criteria
```
□ isEditorOpen exists in stageContext.types.ts
□ isEditorDirty does NOT exist in stageContext.types.ts
□ BuilderPage.tsx uses isEditorOpen (not focusedNodeId) for column width
□ Editor column transitions correctly when a node is opened and closed
□ npm run typecheck passes — 0 errors
```
