## Prior art compliance

The plan names the required prior art: `expired/ui-ux-discovery`, `expired/frontend-discovery`, `completed/ux-discovery-v2`, `completed/frontend-discovery-v2`, and `completed/folder-structure-v2`.

Read expired prior art:
- `expired/ui-ux-discovery` found token drift, dead CSS, typography/radius gaps, and duplicated visual patterns; this draft incorporates that by re-running token/hardcoded-value audits against live `v0.3.5` instead of copying stale counts.
- `expired/frontend-discovery` found component tree, state-flow, and duplication candidates; this draft incorporates that through FP-R3 modularization and churn-risk mapping.

The draft correctly treats completed v2 outputs as prior art to re-verify, not as current counts. It also inherits the folder-structure-v2 canonical homes in the carry-forward contract.

## Handoff quality

R1-R4 outputs are useful inputs for FP-R5: brand token deltas, hardcoded-value baseline, modularization candidates, and finalize criteria have explicit output files. The missing handoffs are decision closure, live-builder interaction evidence, brand/UI interpretation, theme-token direction, homepage/version reference evidence, and implementation routing. Without a decision register, a per-island/per-flow browser-backed gap inventory, a brand-vs-app-UI interpretation note, and a three-family agent/task matrix, FP-R5 can pass by counting open questions and listing generic implementation sprints while still leaving the next plan unclear.

The carry-forward contract exists and every sprint reads and updates it, satisfying `core.md §27` once the write-scope contradiction is fixed.
