# Plan Audit: frontend-polish-v0.3.5

## Verdict

NEEDS REVISION

**Reason:** The FP-R4/FP-R5 redesign is the right direction, but the drafted plan still mixes already-completed discovery sprints with newly executable sprints and does not yet force implementation sprint drafts to be PO-checkable, coverage-complete, and requirement-debt-reducing.

## Prior Art Compliance

The plan correctly incorporates the critical prior art:
- RS-R11's distinction between "0 delivery-confirmed frontend requirements" and "not 0 linked."
- Valid FP-R0/R1/R2/R3 outputs.
- `core-interaction-model.md`.
- The three-family model: `change-token`, `change-component`, `wire-mockup-data`.
- `completed/frontend-discovery-v2`, `completed/ux-discovery-v2`, and `completed/folder-structure-v2` as structural prior art.

The main prior-art risk is not omission; it is lifecycle ambiguity. The plan preserves old FP-R0–FP-R3 sprint files as if they are still executable, even though their outputs are already the prior art that FP-R4/FP-R5 should consume.

## Handoff Quality

FP-R4 is a strong handoff into graph-grounded synthesis: it requires canonical graph IDs, coverage-gap tables, provisional-link labeling, home/version coverage, and the two new requirements.

FP-R5 is almost strong enough, but it needs two more handoff artifacts so executors and the PO do not have to infer:
- **Implementation Coverage Ledger:** FP-R4 criterion → sprint → files likely touched → graph IDs → PO Web Check → gate.
- **Requirement Debt Burn-down:** touched manifestation/link debt → before count → action → after count/evidence.

Without those two artifacts, the implementation plan may still be technically traceable while not being easy for the PO to verify in the website.
