# Plan Audit: backend-discovery-v3

## Verdict

NEEDS REVISION

**Reason:** The plan has the right backend-discovery shape, but activation would hand executors several unverifiable or contradictory gates, especially in BE3-R5's capture workflow.

## Prior art compliance

The plan does read the right prior art and incorporates the important post-refactor shift: `folder-structure-v2` P4 moved app-facing storage services through `apiClient -> mockDispatch`, with 22 routes retained as the mock API seam. It also carries forward BE2's key finding that AI and ClickUp are stubs and that production integration belongs to a follow-up plan rather than this discovery.

The plan improves on expired `backend-discovery` by moving from a manual readiness audit to a dataset plus capture loop. That is the right direction.

What it still risks repeating is BE2's "seam exists but is not enough" problem: BE3-R5 can create scripts and a tap while still failing to prove the live preview capture path. The split recommended above keeps local instrumentation from being mistaken for operational CI capture.

## Handoff quality

R0-R4 have clear enough output destinations and should hand off well after the R1/R3 fixes. R5 is not yet handoff-safe for opencode because it asks the executor to design a capture protocol, a GitHub workflow, artifact extraction, route coverage, scrub semantics, registry patching, and production-flag guard all in one sprint. Splitting R5 will make the handoff much cleaner and prevent a partial CI setup from blocking the whole discovery plan.
