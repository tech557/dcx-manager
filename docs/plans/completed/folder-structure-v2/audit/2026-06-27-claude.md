# Plan Audit: folder-structure-v2

## Verdict

**NEEDS REVISION**

**Reason:** P1 and P2 were audited against the discovery *documents* but not re-verified
against the *current* `src/` tree. Four blocking issues result: P1 migrates token syntax
that has 0 occurrences in code, the largest arbitrary-token group (287 color usages) is
neither scoped nor declared out-of-scope, P2 instructs creating atoms that already exist,
and P2's orphan deletions will break barrel exports and trip P2's own halt rule. All four
are sprint-file fixes (well under an hour each); P3/P4/P5 structure is sound.

> Context vs the prior audit: `audit/2026-06-27-codex.md` returned READY. That audit checked
> the plan against the discovery outputs. This audit re-ran the plan's own `grep`/`ls`
> commands against the live `v0.3.4` tree — which is where the four blockers surface.

## Prior art compliance

The plan complies with `docs/product/decisions/src-structure-decision.md` and the expired
prior art:
- **Tailwind-first, no CSS modules** (UX2-R3 "Rejected approaches") — respected.
- **No full StageContext split; drag-state extraction only** — P3 Step 5 honors this exactly.
- **No mobile breakpoints (1440px–4K)** — P5's 1440/1920/2560 gate honors this.
- **`ai.service.ts` stays a stub; mock retained** — P4 Step 8d/8e honor this; the
  `production-api-client-switch` follow-up is correctly named and fenced off.
- **Context-coupled components stay in place** — P3 Step 5 constraint references the 24
  FE2-R3 components.

**Not reconciled across discovery sources (feeds Blocking #3/#4):** UX2-R3 says *merge*
`StatusBadge/LockBadge/StatusDropdownBadge/PhaseReadinessBadge` into a Badge atom, while
FE2-R3 (which P2 follows) lists `StatusBadge/LockBadge/ReadinessBadge` as *dead orphans to
delete*. Both can be true (the atom already exists, so the old ones are now dead), but the
plan must state that reconciliation rather than leave two contradictory instructions.

`docs/product/requirements/builder/` and `decisions/builder-decisions.md` impose no
constraint this plan violates — it is structure/token cleanup, no feature/behavior change.
There is **no product requirement for a spacing/margin token scale**; the PO's "margins/
spaces" intuition is real in the UI but is currently expressed via native Tailwind scale
(0 arbitrary `m-[var()]`/`p-[var()]`/`gap-[var()]`), so it is out of scope for this plan
unless a new product decision adds a spacing-token requirement.

## Handoff quality

P3/P4/P5 output formats are sufficient for the next agent. P1/P2 are not yet safe handoffs:
an executor following P1 literally will do phantom font/shadow/radius work and report "0
remaining" for patterns that were already 0; an executor following P2 literally will create
a duplicate `TextInput.tsx` and hit a build break deleting barrel-exported files. Both need
the revisions above before the handoff is clean.
