# Agent Skills — DCX Manager

Current as of: 2026-06-26

This document describes the project-specific skill system. Skills reduce token
usage, prevent repeated context loading, and enforce consistent workflows.

---

## Architecture

```
agent-skills/                    ← Canonical, portable skill definitions
├── dcx-sprint-planner/SKILL.md
├── dcx-frontend-refactor/SKILL.md
├── dcx-frontend-verify/SKILL.md
├── dcx-sprint-close/SKILL.md
└── dcx-code-query/SKILL.md

.claude/skills/                  ← Claude Code adapters (thin wrappers + MCP context)
└── dcx-*.md                     ← Generated by sync-skills.sh

.agents/skills/                  ← Other-agent copies (portable content only)
└── dcx-*.md                     ← Generated by sync-skills.sh

scripts/agent/                   ← Deterministic helper scripts
├── code-query.sh
├── verify-plan-state.sh
├── verify-version-state.sh
├── verify-log-claims.sh
├── verify-tooling-state.sh
├── verify-frontend.sh
├── build-current-state.sh
└── sync-skills.sh

docs/generated/CURRENT_STATE.json  ← Non-authoritative snapshot (regenerated)
```

### Canonical vs adapter

The canonical SKILL.md in `agent-skills/` contains portable instructions with
no agent-specific tool names. Claude adapters in `.claude/skills/` reference the
canonical content and add Claude-specific MCP names and deferred-loading guidance.

**Rule:** Never edit `.claude/skills/` or `.agents/skills/` directly.
Edit the canonical `agent-skills/<skill>/SKILL.md`, then run:
```bash
bash scripts/agent/sync-skills.sh
```

## The five skills

| Skill | Trigger | Primary scripts |
|---|---|---|
| `dcx-sprint-planner` | "plan a sprint", "create a sprint for", "draft a plan" | `build-current-state.sh`, `code-query.sh` |
| `dcx-frontend-refactor` | "refactor X", "extract component", "modularize" | `code-query.sh`, `verify-frontend.sh` |
| `dcx-frontend-verify` | "verify my change", "run the tests", "does this pass?" | `verify-frontend.sh` |
| `dcx-sprint-close` | "mark sprint complete", "close the sprint", "sprint is done" | All `verify-*.sh` |
| `dcx-code-query` | "where is X", "who uses X", "find label Y", "consumers of X" | `code-query.sh` |

### Invocation

In Claude Code: skills trigger automatically via description matching. Users can
also invoke explicitly: `/dcx-sprint-planner`, `/dcx-code-query`, etc.

Other agents (Codex, Gemini, opencode): read from `.agents/skills/` at session
start, following any "when to use" description in the SKILL.md frontmatter.

## Deterministic scripts

These scripts exist to replace agent natural-language reasoning with verifiable
shell/Python checks. All output JSON.

| Script | Purpose | Key outputs |
|---|---|---|
| `code-query.sh component <Name>` | Component definition, props, children | `definedIn`, `props[]`, `childComponentsUsed` |
| `code-query.sh consumers <Name>` | All usages of a component | file paths, line numbers |
| `code-query.sh affected <file>` | Reverse dep: who imports a file | file list |
| `code-query.sh labels "<text>"` | UI text label occurrences | file, line, context |
| `code-query.sh hardcoded-tokens` | Hex colors / arbitrary values outside brand/ | file:line hits |
| `code-query.sh duplicate-controls` | Potentially duplicated form controls | grouped by type |
| `verify-plan-state.sh` | Plan folder/frontmatter/status consistency | `pass`, `findings[]` |
| `verify-version-state.sh` | VERSION.md vs package.json vs metadata.json | `versions{}`, `findings[]` |
| `verify-log-claims.sh <log>` | Claimed files exist, deleted files absent, scripts exist | `pass`, `checked`, `findings[]` |
| `verify-tooling-state.sh` | Operational tools, MCP status, code-index freshness | All statuses |
| `verify-frontend.sh` | Run all frontend gates, return JSON | `pass`, `gates{}` |
| `build-current-state.sh` | Generate `docs/generated/CURRENT_STATE.json` | Full state snapshot |

## Token-saving behavior

| Without skills | With skills |
|---|---|
| Agent re-reads AGENTS.md + core.md + all requirements on every session | Skill SKILL.md (~100-200 lines) loaded only when relevant |
| Agent loads full component-usages.json (16,720 lines) to find one component | `code-query.sh consumers X` returns 5-10 focused lines |
| Agent reads all progress logs to find current state | `build-current-state.sh` produces single focused JSON |
| Agent reasons through sprint-close criteria in natural language | `verify-frontend.sh` runs gates deterministically |
| Agent guesses whether files exist | `verify-log-claims.sh` checks filesystem directly |

### Rules for token efficiency

- **Do not read full index files** (`code-index/*.json`) when a `code-query.sh` subcommand covers the question.
- **Do not read all progress logs** when `build-current-state.sh` provides the latest log pointer.
- **Do not run all verification tools** for every change — use the proportional matrix in `dcx-frontend-verify`.
- **Do not load MCP tool schemas eagerly** — use ToolSearch to load only the schemas needed for the current task.

## External-skill review policy

Before installing any skill from a public source (GitHub, community repos):

1. Read its `SKILL.md` completely.
2. Check what tools it accesses (Bash, Edit, network tools).
3. Check what environment variables or secrets it reads.
4. Check what files it can modify.
5. Check its license and maintenance state.
6. Test in a dry-run or isolated environment before activating.

**Do not install generic mega-collections of skills.**
Only install skills with a clear, narrow purpose that this project actually needs.

## Adding or updating a skill

1. Edit `agent-skills/<skill>/SKILL.md`
2. Keep SKILL.md under 500 lines; move large reference material to `agent-skills/<skill>/references/`
3. Run `bash scripts/agent/sync-skills.sh` to update all adapters
4. Test triggering: write 3 representative prompts and confirm the skill triggers correctly
5. Test false-triggering: write 3 prompts that should NOT trigger the skill
6. Update this document if the skill's scope changes

## Portability across agents

The canonical `agent-skills/` content is portable because:
- No Claude-specific MCP names (playwright, context7, chrome-devtools) in canonical SKILL.md
- No Claude hook names or Claude SDK syntax
- Deterministic scripts use bash + Python (available everywhere)
- `.agents/skills/` receives the canonical content directly

To verify portability: a non-Claude agent reading `.agents/skills/dcx-sprint-planner.md`
should be able to follow the instructions using only bash and its available tools.

## Skill evaluation

To evaluate a skill with Skill Creator:
1. Invoke `anthropic-skills:skill-creator` with the skill path
2. Provide 3 representative test prompts
3. Review triggering accuracy and output quality
4. Run description optimization if triggering is inconsistent

See Skill Creator documentation for the full eval loop.

## Current-state snapshot

```bash
bash scripts/agent/build-current-state.sh
```

Produces `docs/generated/CURRENT_STATE.json` with:
- Repository and package versions
- Active plans and sprint files
- Latest progress log
- Available scripts
- MCP operational/awaiting status
- Code-index freshness
- Documentation contradictions
- Git state

**This file is non-authoritative.** For decisions about version, plans, or requirements,
always read the primary sources (`docs/VERSION.md`, `docs/plans/active/`, etc.).

## Third-party skills

### `impeccable` — ENABLED, BRAND-SYSTEM ONLY (quarantine lifted 2026-06-28 by PO)

A third-party design skill (impeccable.style, `pbakaus/impeccable`) installed via `npx skills add`.
It lives in `.agents/skills/impeccable/` (symlinked into `.claude/skills/`). It is **not** managed by
`sync-skills.sh` (that only handles the `dcx-*` list).

**Quarantine history:** installed 2026-06-28 and quarantined until `folder-structure-v2` completed.
That plan completed 2026-06-28 (now in `docs/plans/completed/folder-structure-v2/`), so the PO lifted
the quarantine: the `PreToolUse` `Skill` guard hook was removed from `.claude/settings.json` and
`skillOverrides.impeccable` set to `"on"`. The guard script `scripts/agent/guard-impeccable.sh`
remains in the repo but is **dormant** (no hook references it).

**Standing governance (still enforced by policy, not a hook):**
- **EDITS ARE BRAND-SYSTEM ONLY.** impeccable may *edit* only `src/brand/` (tokens, design system). It
  must **NOT** edit component or application code (`src/ui`, `src/builder`, logic, services); component
  modularization is `dcx-frontend-refactor`'s job. Two explicit modes (see the `frontend-polish-v0.3.5`
  plan): **`impeccable-brand-audit`** edits/recommends `src/brand/` only; **`impeccable-visual-review`**
  may *inspect* the running UI (screenshots) read-only and write markdown findings — never edits code.
  (No `src/components/` exists post folder-structure-v2.)
- **Run `/impeccable init` first** so it reads the existing `src/brand/` system (committed brand
  colors → identity-preservation wins) instead of imposing a generic palette.
- **Brand source of truth for corrections:** `brandbook.pdf` (repo root). When reconciling the design
  system, brandbook values override current tokens; apply corrections as token edits in
  `src/brand/styles/tokens.css` / `src/brand/tokens.ts` only.
- **Security:** the installer rated it `Med Risk` (Gen/Snyk); skills run with full agent permissions.

**First sanctioned use:** the `frontend-polish-v0.3.5` discovery plan (sprint FP-R1). See
`docs/plans/drafted/frontend-polish-v0.3.5/`.

**To re-quarantine** if ever needed: re-add the `PreToolUse` `Skill` hook (it re-activates the dormant
guard script) and set `skillOverrides.impeccable: "user-invocable-only"`.
